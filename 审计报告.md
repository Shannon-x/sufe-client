# Xboard-Client 首席架构师审计与优化报告

## 1. 总体评估

这个客户端**是一个"功能骨架完整、但三大闭环均存在阻断级缺口"的半成品**，桌面端最接近可用、移动端结构性不可用。桌面端的登录/购买/连接主路径在"理想条件"（无 captcha、订阅有效、端口空闲、不崩溃）下能跑通，工程质量也不低（重连退避、健康心跳、降级到系统代理等都有设计），但**一旦遇到真实世界的常见状态（Cloudflare Turnstile、订阅过期、端口被占、内核崩溃）就会静默失败或"假连接"**。最致命的共性问题是**"失败被当成功"**：订阅 403 被写成空配置后仍报"已连接"、内核崩溃后系统代理残留指向死端口却无人清理——用户看到的是"连上了但全程断网"，且无法自助恢复。移动端（iOS/Android）的隧道数据面尚未打通（Android 从不注入 TUN fd、iOS 入站包全丢），属真正的未实现功能。此外存在**发布级安全漏洞**：提权 helper 不校验配置/日志路径且承诺的 IPC HMAC 完全没实现，构成本地提权与任意 root 文件写。核心短板可归纳为：**(1) 错误路径处理缺失（4xx/崩溃/端口冲突）、(2) 三大诉求的"最后一公里"未闭环（自动重连、内购回流、自动续登）、(3) 移动端数据面与提权安全是硬伤**。

## 2. 三大诉求专项分析

### 2.1 登陆即可用

**现状**：桌面端交互式登录后会写 `sessionStorage['xboard.autoConnectAfterLogin']='1'`（Login.vue:103-106），Home 挂载时 `autoConnectAfterLogin()`（Home.vue:646-662）读取标志并在 `canConnect` 通过时自动连接——首次登录的"登录即连"是成立的，且有正确的 `canConnect` 门控（Home.vue:654）避免对无套餐用户发起注定失败的连接。登录态持久化、冷启动恢复（auth.ts:34-52）、24h 离线容错（session.rs:125-142）也都做了。

**关键差距**：
1. **(blocker) captcha 字段名全错位**——`auth.rs:15/31/91` 把 Turnstile 序列化成 `cf-turnstile-response`，但后端 `CaptchaService` 读的是 `turnstile_token`（CaptchaService.php:41）；reCAPTCHA v3 应走 `recaptcha_v3_token`（:68）但客户端从未定义该字段；`send_email_verify`（auth.rs:52-69）只塞 `recaptcha_data`。**在 Cloudflare Turnstile（Xboard 默认推荐）部署上，客户端内注册与发送邮箱验证码恒定 400，开户链路被完全阻断**。注释 auth.rs:47-51 还写反了后端行为。
2. **(blocker) 订阅 403 静默化（见 2.3）** 直接破坏过期/欠费用户的"登录即可用"——登录成功但点连接得到假"已连接"。
3. **(minor) 老用户重开 App 不自动重连**——冷启动 `bootstrap()`（auth.ts:34-52）恢复会话后不设 autoConnect 标志，最常见的"老用户重开"场景仍需手点一次连接。
4. **(minor) hydrate 后 IPC 异常误清会话**——auth.ts:43 `api.checkLogin().catch(() => false)` 把任何 IPC 层异常（非鉴权失败）折叠成 false 并清掉刚恢复的会话（auth.ts:44-45），与 Rust 侧 24h 容错相矛盾。
5. **(minor) 缺免密/快速登录**——`token2Login`/`getQuickLoginUrl`/`loginWithMailLink` 全未实现（auth.rs 仅 login/register/forget/sendEmailVerify），无法做邮件魔术链接/扫码续登。

**落地方案与改动点**：
- captcha 字段对齐（**P0**）：`core/src/api/auth.rs:15-19, 31-35, 91-95`——删除 `cf-turnstile-response` rename，按 captcha_type 动态选字段：turnstile→`turnstile_token`、recaptcha-v3→新增 `recaptcha_v3_token`、recaptcha-v2→`recaptcha_data`；`send_email_verify`（auth.rs:52-69）的 Body 同样补 `turnstile_token`+`recaptcha_v3_token` 并按类型选填。captcha_type 从 `guest/comm/config` 拉取。前端 Login.vue:96 / Register.vue:209 停止把 v3 token 合并进 recaptcha 字段。
- 冷启动自动重连（**P1**）：`desktop/src/stores/auth.ts:34-52` 在 `bootstrap()` 成功且 `checkLogin` 通过、且 subscribe 有效时设 autoConnect 标志（做成"尊重上次手动断开"的开关）。
- 修 IPC 误清（**P1**）：auth.ts:43 区分"明确登出（false）"与"IPC 异常（保留 session）"，异常分支不清 session。
- 免密登录（**P2**）：`core/src/api/auth.rs` 新增 `token2_login(verify)`（GET query，裸 `{data:{...}}` 单独解析）；接 `lib.rs` 已注册的 deep-link 插件解析面板 `/#/login?verify=...`。

**验收标准**：在 captcha_type=turnstile 与 recaptcha-v3 的面板上，客户端内能完成"发码→注册→自动登录→自动拉订阅"；老用户重开 App 在有有效订阅时自动重连（开关可关）；token 被服务端吊销后冷启动不会误清本地会话残留。

### 2.2 客户端直接购买

**现状**：桌面端 PurchaseModal 已实现 套餐→优惠券预览→下单→checkout→轮询 的完整骨架；type=0（AlipayF2F 当面付）渲染二维码、type=-1（免费/余额抵扣）走结算分支都正确；下单撞到"存在未完成订单"时有反应式 `saveOrderWithPendingHandling`（PurchaseModal.vue:454-557）自动 cancel 重试。这是三大诉求里完成度最高的。

**关键差距**：
1. **(major) 付费支付跳系统浏览器，无法 app 内闭环**——type=1（Mgate/Epay/Coinbase/Btcpay 等主流跳转网关）走 `shellOpen(resp.data)`（PurchaseModal.vue:426-434）在外部浏览器打开；后端 `return_url` 又写死为网页 hash 路由 `/#/order/{trade_no}`（PaymentService.php:85），支付完成落到面板网页、不回 app。用户付完款要手动切回 app 等轮询。这是"客户端直接购买"体验断点的最大缺口。
2. **(minor) 免费/全额抵扣单在无支付方式时被 UI 卡死**——提交按钮 disabled 条件含 `methods.length === 0 || selectedMethod == null`（PurchaseModal.vue:938-949），且 submit 开头强制要求选 method（:363-374）；当订单被余额/100%券抵到 0 且面板无 enable 支付方式时永远点不了提交，无从触发后端免费分支。
3. **(minor) 轮询无超时/无放弃指引**——`startPolling` 每 4s 一次无上限（PurchaseModal.vue:561-615），用户中途放弃或 notify 久不到时 modal 停在不可关闭的 pending 态。
4. **(minor) 展示价未含手续费**——`effectivePriceCents` 只算"基价−券"（PurchaseModal.vue:146-157），不加 handling_fee；后端 checkout 会按 `round(total*percent/100)+fixed` 多扣（OrderController.php:144-147），实付高于显示。
5. **(minor) coupon/check 漏传 period**——`check_coupon` 只发 `{code, plan_id}`（order.rs:225-240），周期限定券预览通过但 order/save 时被后端按真实 period 拒。
6. **(minor) 复制链接用 navigator.clipboard**——PurchaseModal.vue:709 / Home.vue:472 在 WKWebView/WebView2 自定义协议下可能静默失败却提示"已复制"。

**落地方案与改动点**：
- app 内嵌支付窗（**P1**）：`desktop/src/components/PurchaseModal.vue:426-434` 对 type=1 改用 Tauri `WebviewWindow` 打开支付 URL，监听导航到含 `trade_no` 的 return_url 时关窗并立即触发一次 `check_order`；保留 `shellOpen` 作降级。或接 deep-link 让后端提供 app 深链 return_url。
- 免费单跳过选支付（**P1**）：PurchaseModal.vue:363-374, 938-949——当 `effectivePriceCents===0` 时允许 selectedMethod 为空提交，checkout 传占位 method（后端免费分支忽略），按 type=-1 结算。
- 轮询超时（**P2**）：PurchaseModal.vue:561-615 加总时长上限（5-10min）/退避，超时切到"仍未确认支付"态并给"重新检查/取消订单/订单页查看"入口。
- 手续费并入合计（**P2**）：PurchaseModal.vue:146-157 选中 method 后追加 `round(total*percent/100)+fixed`。
- coupon period 透传（**P2**）：`core/src/api/order.rs:225-240` + `api.ts:138` + PurchaseModal.vue:276-300 把 `periodKey`（legacy key）一并传入。

**验收标准**：付费跳转类网关能在 app 内嵌窗完成支付并自动回流轮询；余额/全额券抵到 0 的订单（含无支付方式面板）能一键开通；顶部合计金额与最终扣款一致；周期限定券预览与下单结果一致。

### 2.3 连接可用性

**现状**：桌面端有完整的 TUN-first 连接状态机、退避重试（manager.rs:74）、心跳/流量双探活（manager.rs:312-352, 600-626）、提权失败降级系统代理、disconnect 时先清代理后停内核（manager.rs:689-713）。设计意图是好的，但**错误与崩溃路径大量缺失**，导致"成功率/可恢复性/退出清理"三项都有阻断级问题。

**关键差距（按严重度）**：

1. **(blocker) 订阅 403/4xx 被当成功→"假连接"全程断网**——`fetch_subscribe`（client.rs:40-58）捕获了 status 但无条件 `Ok` 返回，无 `error_for_status`；`fetcher.fetch`（fetcher.rs:51-77）只特判 304，403+空 body 会落到 fetcher.rs:70 `tokio::fs::write` 把**空 body 覆盖写入缓存**（连之前的有效缓存也被污染）；`patch_mihomo("")` 走空 Mapping 分支（inject.rs:56）产出无 proxies 仅 `MATCH,GLOBAL` 的配置；内核照常 spawn、driver attach 成功，最终 `publish(Connected)`（manager.rs:547）。**任何到期/欠费/封禁用户首连即触发：UI 显示"已连接"但无可用出口（TUN 下流量丢弃或裸 DIRECT 泄漏），且不提示续费、不可恢复**。manager.rs:372 注释声称"对 4xx 快速失败"与实现不符。同问题在 `preview_subscribe_nodes`（nodes.rs:58）独立复现一次：把"订阅不可用"当"0 个节点"，侧栏只显示通用空态无续费引导。

2. **(blocker) 提权 helper/svc 安全漏洞**——`start_kernel`（helper:235-270 / svc:500-541）只 `validate_exec_path(exec_path)`（helper:242），对 `work_dir`/`cfg_path`/`log_path` **不做任何路径白名单校验**，直接 `create_dir_all(work_dir)`、以 root `OpenOptions create+append` 打开 `log_path`（helper:249-252）、以 `-f cfg_path` 加载任意配置（helper:258-259）。模块文档（helper:22-29、ipc.rs:9-15）承诺的 per-install HMAC **完全没实现**（`HELPER_SECRET_PATH` 仅被 no-op 引用）。macOS 套接字 root:staff 0660 且无 peercred 校验，任意 staff 组本地用户均可连接 → **以 root 任意文件写（覆盖 /etc/* 提权）+ 任意 mihomo 配置（监听/DNS/路由/external-ui，等价代码执行）**。Windows svc 有逐连接 SID 校验，故限定为"安装用户从普通权限提权到 LocalSystem"。

3. **(blocker, 移动端) Android TUN fd 从不注入**——`ConnectionManager` 把 `tun_delegate` 存进未用的 `_tun_delegate`（ffi/manager.rs:40），connect 路径不消费它（注释自承"the launcher is currently always DirectLauncher"）；移动端 `ensure_privileged` 命中 `cfg(not(...))` 直接 Ok（launcher.rs:303），final_mode=Tun，patch 出 `auto-route:true / device:<name>` 让 mihomo 自己开 /dev/net/tun——Android 沙箱无 CAP_NET_ADMIN，**`establishTun` 永不被调用，连接结构性不可能成功**。

4. **(blocker, 移动端) iOS 入站包全丢 + 假已连接**——`writeIntoBox`（PacketTunnelProvider.swift:148-159）是显式 no-op（line 158 `_ = packet`），从系统读到的上行包从不喂进 sing-box，**隧道建立但承载不了任何流量**；且 `AppModel.connect` 在 `startVPNTunnel` 返回后立即标记 `.connected`（AppModel.swift:329）不校验 NEVPNStatus，失败不可见。另：sing-box 注入引用 `geosite-cn` 等 rule_set 但从不输出 `route.rule_set[]` 定义块（inject_singbox.rs:198-209, 624-649），sing-box 1.8+ 启动即 `rule-set not found` 拒绝运行。

5. **(blocker) 内核崩溃(Exited)无清理→系统代理残留指向死端口**——supervisor 收到 `Exited` 仅 `health_tx.send` 后 `return`（manager.rs:632-647），不改 state、不 `proxy_setter.clear()`、不 take handle；前端 kernel-failure 监听器（connection.ts:114-146）只设 error 态不调 disconnect。**SystemProxy 降级模式下 mihomo 崩溃后系统代理仍 enable 指向死端口，全系统断网且无自动清理**，等同"应用把网弄坏了"。注意 macOS/Windows 走 Unresponsive 路径，仅修 Exited 不够（manager.rs:643-646 注释"manager owns recovery"与无自动重连的实现不符）。

6. **(major) 节点选择不持久化**——`patch_mihomo` 不写 `profile.store-selected`/`cache-file`（inject.rs 全程无），sing-box 显式 `cache_file.enabled: false`（inject_singbox.rs:192）；select_proxy 只是内存态 PUT /proxies；每次 disconnect+connect（含换模式 connection.ts:201）重启内核，**手选节点丢失回退组默认**——直接破坏"节点切换稳定"。

7. **(major) 控制器端口 9090/混合端口 7890 写死无冲突检测**——manager.rs:48-49 编译期常量，与 clash-verge 等共存时 `wait_for_controller` 去 poll 异己控制器、secret 不匹配 401 耗满 5s 慢失败；7890 被占时系统代理指向别的应用→流量错转/断网。

8. **(major) connect 命令跨 await `.unwrap()` panic**——connection.rs:34 snapshot_auth 后 :42 await user_subscribe，:45 `state.auth.write().as_mut().unwrap()`；期间登出会在 None 上 panic，**Tauri 命令线程 panic 使进程异常终止且不走清理路径，TUN/系统代理残留**。

9. **(major) 退出清理仅靠 WindowEvent::Destroyed**——唯一清理路径是 lib.rs:164 的 `block_on(km.disconnect())`；托盘退出 `app.exit(0)`（lib.rs:125）、ExitRequested（lib.rs:230）不 disconnect；`KernelManager` 无 `impl Drop`。panic/SIGKILL/Destroyed 未触发时系统代理与 TUN（及独立特权进程的 mihomo）全部残留。

10. **(major, CI) 内核与 wintun.dll 下载零校验**——install-mihomo-sidecar.sh:75-89 等三脚本裸 curl 解压打进签名包，`mirror-mihomo.sh` 算了 sha256 却从未被 workflow 引用；上游被篡改/CDN 劫持会把恶意代理内核打进安装包（注意：运行时 kernel.rs:60-107 的 ed25519+sha256 更新校验是完整的，缺口仅在 build-time bundling）。

11. **(minor 系列)**：connect 无重入保护双击会孤儿化旧 handle 并经 5s 无谓重试（connection-kernel-6）；硬 SIGKILL 不给 mihomo 回收 Windows strict-route 防火墙规则的机会（connection-kernel-5）；Unresponsive 仅上报不自愈（connection-kernel-9）；崩溃后启动不做 stale-proxy 清理（profile-tunnel-4）；`default-nameserver` 注入非 IP 关键字 `system` 可能被部分 mihomo 版本拒（profile-tunnel-6）；连接中无界面取消入口（frontend-ux-5）；macOS 默认 TUN 首连即弹管理员授权弹窗（tauri-shell-3）；node_geo 自动上报真实节点 IP 给 ip-api.com 且临时切换实时选择（critic-3）；`allow-lan` 被硬编码 false 覆盖订阅（critic-8）；XrayDriver 全 stub（connection-kernel-10）。

**落地方案与改动点**：
- 订阅 4xx 处理（**P0**）：`core/src/api/client.rs:40-58` 对 `status >= 400`（尤其 403）映射为 `XboardError::SubscriptionUnavailable`；`fetcher.rs:62-70` 收到错误**不写盘、不覆盖既有缓存**、向上传播；`manager.connect`（manager.rs:431-440）在 patch 后校验配置至少含一个非空 `proxies`，否则 fail；`nodes.rs:58` 据 403 返回 `subscription_unavailable` 让 NodeRail 显示续费引导。
- 提权安全（**P0**）：helper/svc `start_kernel` 对 `cfg_path/work_dir/log_path` canonicalize 后强制校验位于调用用户 Application Support 子目录（macOS 用 LOCAL_PEERCRED 解 uid、Windows 用调用方 SID）；真正实现 IPC HMAC（用 `HELPER_SECRET_PATH` per-install 密钥对 Frame 验签）或删除误导性安全注释并以严格路径白名单兜底。
- 崩溃即清代理（**P0**）：`core/src/kernel/manager.rs:632-647` 对 Exited **和**不可恢复 Unresponsive 都 take handle + 若 SystemProxy 则 `setter.clear()` + publish Error/Disconnected；或前端 connection.ts:114-146 kernel-failure(exited) 强制调一次 `api.disconnect()`（两端二选一确保 OS 代理回收）。
- 移动端数据面（**P0**，见 §5 阶段3）：Android 新增 mobile Launcher，Spawning 前调 `tun_delegate.establish_tun` 拿 fd 注入 `tun.file-descriptor` 并去掉 auto-route/device；iOS 把 `writeIntoBox` 接到 Libbox 真实写入（或 openTun 返回真实 utun fd），订阅 NEVPNStatusDidChange 驱动状态，sing-box 补 `route.rule_set[]` 定义。
- 节点持久化（**P1**）：inject.rs 注入 `profile: { store-selected: true }` + 稳定 `cache-file` 路径；inject_singbox.rs:192 改 `enabled: true` 并给持久 path；或 manager 侧记录 last_selected 在 connect 成功后重放 select_proxy。
- 端口冲突检测（**P1**）：connect 前用 `TcpListener` 试绑选空闲控制器/mixed 端口，写进 patched YAML 并传给 driver/proxy_setter；`wait_for_controller` 命中 401 快速判定端口被占。
- connect unwrap（**P1**）：connection.rs:45 改 `if let Some(a) = state.auth.write().as_mut() { a.subscribe_token = ... }`。
- 退出主清理（**P1**）：lib.rs RunEvent::ExitRequested 显式带超时 disconnect；为 KernelManager 加 Drop 兜底或 helper/svc 加"父进程消失即自毁"看门狗。
- CI 哈希钉死（**P1**）：三个 install 脚本对下载归档做 sha256 比对（消费 mirror-mihomo.sh 的 manifest.json 或硬编码每版本预期哈希），失败即非零退出。

**验收标准**：过期用户连接得到"订阅不可用/请续费"而非假"已连接"；普通 staff 用户无法通过 helper 以 root 写任意文件；内核崩溃后系统代理被自动清除、网络恢复；端口被占时秒级明确报错而非 5s 超时；换模式/重连后手选节点保持；Android/iOS 隧道能实际承载流量；退出（含托盘退出/崩溃）后系统设置无残留代理。

## 3. 后端对接问题清单（客户端↔Xboard-sh 不匹配项）

## 6. 风险与注意事项

1. **"真 bug" vs "未实现功能"务必区分对待**：订阅 4xx 静默化、connect unwrap panic、崩溃不清代理、helper 路径未校验、captcha 字段错位、节点选择不持久化、端口写死——这些是**桌面端已上线路径的真 bug**，修复成本低、收益高，应优先。而 Android TUN fd / iOS 入站写入 / token2Login / Android updater / Xray driver / 内核自更新——是**未实现功能**，工作量大，可按阶段3 排期，发布说明里诚实标注移动端为 beta。
2. **后端响应信封不统一是最大对接坑**：`token2Login(verify)` 裸 `{data:{...}}`、`checkout` 裸 `{type,data}`、`server/fetch` 裸 `{data:[...]}`、订阅端点裸 YAML/403 空体——任何新接客户端方法都要单独确认解析路径，不能套标准信封。
3. **修订阅 4xx 时勿误伤离线兜底**：fetcher 的 `cache_fallback`（网络失败用旧缓存）是有价值的，**只对 4xx 业务失败不写盘**，网络层失败仍走 cache_fallback，二者要分流。
4. **提权安全修复需双端 + 跨平台验证**：macOS 用 LOCAL_PEERCRED、Windows 用 SID，路径白名单要 canonicalize 后比对避免符号链接 TOCTOU；HMAC 若实现要保证 helper 与 core 双端密钥一致。
5. **panic=abort 会放大所有 panic**：在修 connect unwrap 的同时，建议全面 grep FFI 边界的 `unwrap/expect`，并对移动 cdylib 切 `panic=unwind`，否则任一边界 panic 直接闪退而非可恢复错误。
6. **mihomo 配置改动需对照目标内核版本**：`default-nameserver` 的 `system`、`store-selected`、`tun` 各键在 v1.18.7 与 v1.19.x 行为可能不同；UA 版本号当前与 MIHOMO_VERSION 解耦，升级内核时注意一并验证订阅协议匹配（'meta' 子串命中，版本号不参与）。
7. **发布管线 hands-off 前置**：keystore secret、gh-pages 并发、updater 第二 endpoint 这三处不修，v0.1.0 自动发布大概率在某一平台清单或 APK 签名步骤失败，与 MEMORY 中"hands-off 发布"诉求直接冲突，应纳入阶段1/2 的发布门禁检查清单。
